- 'Architecture.txt' : 기능 구성 상세.
- 'SlideGuide.txt' : 슬라이드 제작 규칙.

[AI 프롬프트 지침]
전체 'markdown' 문서를 분석한 후, Architecture.txt의 내용을 SlideGuide.txt의 규칙에 따라 슬라이드 제작. ProjectInfo.txt는 필요시 참조.
단계: 1. 문서 분석 → 2. 페이지 작성 계획 → 3. 슬라이드 파트별 생성 → 4. 중복/누락 검토 → 5. 필요시 목차, 개요 갱신.

[슬라이드 작성 정책]
- 개발 계획, 일정/문서 구조/기초 설명/마케팅/사업/운영 비전, 기술 스택(API/DB), QA/테스트 계획 작성 불필요.
- 주제에 해당하는 U I, 와이어프레임의 명확한 구조 설계에 집중.
- 페이지 수: 한 주제의 슬라이드 전체를 최대 15페이지 이하로 작성.
- 페이지간 반복, 중복적인 내용이 없도록 함.
- 페이지 레이아웃: 페이지 구분 명확, 헤더/푸터 겹침 방지. 세로폭 유동적, 빈 공간 최소. 2개 이상 와이어프레임 배치시 가로 혹은 바둑판 배치.
- 와이어프레임: 목업 렌더링. 모바일 디바이스 표현시 화면비율(19.5:9) 유지. 
- 와이어프레임의 디자인 일관성 유지 (앱: Flutter Material, 웹: Ant Design).
- 와이어프레임의 화면비율 반응형 고려 (모바일: 세로, 웹: 가로). 
- 와이어프레임 별 상황에 맞는 헤더, 네비게이션바를 항상 적용.
- 와이어프레임에 렌더링한 기능/작동 데이터 설명: 각 와이어프레임의 우측이나 하단에 정리. 각 메타 정보, 호출/참조되는 데이터를 설명.
- 모달/팝업: 개별로 예시 디자인을 배치하여 설명. 
- 탭: 탭 입력 상황별 개별 와이어프레임 구분하여 설명.
- 와이어프레임 내 스크롤/슬라이드 작동하도록 구현.
- 와이어프레임 내 버튼들의 입력 피드백 작동하도록 구현. 
- 슬라이드 작성 중 진행 상태 설명하는 등의 페르소나 생략.

[슬라이드 구성 정책]
- 푸터: 좌상단 "ⓒ2025 BESoft Corporation.".
- 표지: 제목 '{프로젝트 이름}:{앱 이름}', 아래 중앙 'BESoft. {연도} by JK'.
- 목차: 전체 내용 고려, 표지/목차 포함 넘버링. 각 페이지의 간단 개요 설명.
- 플로우: 문서에서 다루는 핵심 내용에 대한 사용자 타입별 여정 시나리오.
- 와이어프레임 UI 및 내부 기능별 상세 정의: 사용자 타입별 CRUD 기준 사용자 여정에 맞는 순차적 와이어프레임 구성, 각 기능 및 작동 항목들의 상세한 설명, 사용되는 메타/데이터 정보들의 설명.
